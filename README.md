# zeroTrace: A Lightweight Dynamic probe for User Space

## 1. 项目目标

本项目旨在实现一个纯用户态的动态追踪框架。区别于基于软中断陷入内核的 `uprobe`，本项目预计通过 `ptrace` 实现**动态二进制插桩技术**，在不进入内核态的前提下，实现对目标进程指定函数的参数提取和返回值捕获。

## 2. 系统模块划分

系统在逻辑上严格分为两个运行平面：

### 2.1 控制平面（注入器 Injector）

作为一个独立进程运行，负责对目标进程进行环境初始化和代码注入。

- **进程控制**：通过 `ptrace` 附加并暂停目标进程及其所有线程，保障修改内存时的并发安全。
- **内存分配**：控制目标进程执行 `mmap` 系统调用，分配具备可读可执行权限的内存，用于存放探针跳板代码。
- **指令解析与重定位**：利用反汇编引擎（如 Capstone），精确计算目标函数开头需覆盖的指令长度（避免截断变长指令），并处理相对寻址指令的偏移量修正。
- **代码植入**：将预编译好的 Stub 写入目标进程，并将目标函数开头的指令原子性地替换为无条件跳转指令，跳转到植入的 Stub 中。

### 2.2 数据平面（Payload - 探针载荷）

运行在目标进程空间内，采用“微型跳板（Thunk）+ 共享枢纽（Generic Stub）”的高度解耦设计,相关代码见[stub.md](./stub.md)：

- **微型跳板 (Thunk)**：为每个 Hook 点生成极简的代码片段，仅负责压入“探针元数据指针”并跳转至通用枢纽。
- **通用处理枢纽 (Generic Stub)**：全局唯一的汇编逻辑，负责全量上下文的保存与恢复。
- **影子栈 (Shadow Stack)**：在内存中维护线程私有的返回地址栈，支持**函数递归调用**与**多层嵌套 Hook** 的精准回溯。

## 3. 核心执行流 (Execution Flow)

### 阶段一：入口劫持与影子栈部署 (Entry Phase)

1. **触发跳转**：目标函数入口执行 `jmp thunk`，随后通过 `call` 进入全局 `Entry_stub`。此时栈顶自动压入**原指令区（Detour）**地址。
2. **上下文深度保护**：
   - 压栈保存 $16$ 个通用寄存器及 `RFLAGS` 标志寄存器（共 $128$ 字节）。
   - **ABI 栈对齐**：通过非易失性寄存器（如 `r12`）备份 `rsp`，强制进行 **16-byte alignment**，确保后续 C 探针逻辑兼容 SSE/AVX 指令集。
3. **返回地址劫持**：
   - 从栈中提取 `Original_Ret_Addr`，压入用户态**影子栈（Shadow Stack）**。
   - **篡改返回路径**：将原程序栈帧中的返回地址覆盖为全局 `Return_stub` 的地址。
4. **精确接续**：通过 `ret` 指令弹出 `Detour` 地址，执行被搬运的原始指令后 `jmp` 回原函数主体。

### 阶段二：返回捕获与路径复原 (Return Phase)

1. **重入陷阱**：函数执行 `ret` 时，由于地址已被劫持，控制流强行进入 `Return_stub`。
2. **返回值提取**：在保存上下文后，直接从 `rax` 寄存器中提取函数执行结果，传递至用户回调。
3. **路径归还**：
   - 从影子栈中弹出该线程对应的 `Original_Ret_Addr`。
   - 恢复所有寄存器状态，利用 `ret` 弹出真实返回地址，实现对原调用者的无感回归。
